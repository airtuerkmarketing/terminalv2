## 🔴 BUGS (Render-Fehler)

### BUG-1 — Master-Deck-Cards: doppeltes Sprach-Suffix „(DE)DE" / „(EN)EN"
- **Schweregrad:** 🔴 Bug (kosmetisch, aber sichtbar auf jeder Brand-Seite mit Master Deck)
- **Datei:** `src/components/blocks/document-list.tsx:60-63` (Funktion `PreviewCards`)
- **Code-Zitat:**
  ```tsx
  <a href={d.href ?? "#"}>
    {d.title}
    {d.lang ? <span className="lang">{d.lang}</span> : null}
  </a>
  ```
- **Live-Beleg (Production /service-center-antalya):** `airtuerk Master Deck (DE)<span class="lang">DE</span>`
- **Ursache:** Die geseedeten `document_list`-Items tragen den Sprach-Code **doppelt**: einmal im `title` („airtuerk Master Deck **(DE)**") und einmal im `lang`-Feld („DE"). Der Renderer zeigt beide → „…(DE) DE".
- **Betroffen:** `/airtuerk-service/master-deck`, `/service-center-antalya/master-deck` (beide nutzen `preview_cards` mit „(DE)"/„(EN)"-Titeln + lang-Feld). Quelle: Migrationen 0012, 0022.
- **Fix-Vorschlag (eine von zwei Varianten — Buhara entscheidet):** (a) `lang` aus den Seed-Daten entfernen (Title trägt die Sprache schon), oder (b) das „(DE)" aus dem Title entfernen und nur den `lang`-Badge zeigen. Variante (b) ist konsistenter mit dem Badge-Pattern.
- **Aufwand:** S (Daten-Migration ODER Renderer-Zeile)

### BUG-2 — Letterhead-Cards: ZIP-Dateien tragen „DOC"-Badge
- **Schweregrad:** 🔴 Bug (falsches Dateiformat-Label)
- **Datei:** `src/components/blocks/document-list.tsx:26-28` (`ft()`) + `:34-39` (`ListRows`)
- **Code-Zitat:**
  ```tsx
  function ft(d: DocumentItem) {
    return (d.filetype ?? "doc").toUpperCase();
  }
  ...
  <span className={`ft ${d.filetype ?? ""}`}>{ft(d)}</span>
  ```
- **Live-Beleg (Production /service-center-antalya):** `<span class="ft ">DOC</span><span class="info"><span class="name">Letterheads — Hauptkonto</span>` — und der Meta-Text sagt „ZIP". Auf der Seite erscheint „DOC" **2×** (beide Letterhead-ZIPs).
- **Ursache:** Die Letterhead-`document_list`-Items haben **kein `filetype`-Feld** (nur `meta: "ZIP"`). `ft()` fällt auf Default „doc" → Badge „DOC", obwohl es ZIPs sind. Zusätzlich erzeugt `ft ${d.filetype ?? ""}` die Klasse `ft ` mit **leerem Modifier** → kein Farb-Styling (`.ft.pdf`/`.ft.word` greifen nicht).
- **Betroffen:** `/airtuerk-service/letterhead`, `/service-center-antalya/letterhead` (beide `list_rows`). Quelle: Migrationen 0012, 0022.
- **Fix-Vorschlag:** `DocumentItem.filetype` um `"zip"` erweitern (Schema in `src/lib/blocks/schemas.ts` + types) und Letterhead-Seeds `filetype:"zip"` geben; ODER `ft()` aus `meta` ableiten wenn `filetype` fehlt. Erste Variante sauberer. **Hinweis:** das ist eine Schema-Erweiterung → eigener kleiner Migrations-/Code-Schritt.
- **Aufwand:** M (Schema + Renderer + Daten)

### INK-1 — Download-Affordances „mal rund, mal eckig" (Buhara bestätigt)
- **Schweregrad:** 🟡 Inkonsistenz (uneinheitliches Interaktionsmuster)
- **Befund (Tabelle, alle mit Datei:Zeile + CSS-Regel):**

  | Affordance | Verwendung | border-radius | Form |
  |---|---|---|---|
  | `.card-dl` | document-list.tsx:54, logo-grid.tsx:14 | `var(--radius-full)` (blocks.css:138) | **rund** (34px Kreis) |
  | `.dl-pill` | asset-block.tsx, logo-showcase.tsx:26 | `var(--radius-full)` (blocks.css:156) | **rund** (Pille) |
  | `.al-dl` | asset-library.tsx:148 | `var(--radius-sm)` (asset-library.css:34) | **eckig** (rounded-rect) |
  | `.doc-download` | document-library.tsx:188 | `var(--radius-sm)` (document-library.css:92) | **eckig** |
  | `.doc-row-dl` | document-library.tsx:226 | `var(--radius-sm)` (document-library.css:68) | **eckig** |
  | `.doc-dl-badge` | document-library.tsx:195 | `6px` **hardcodiert** (document-library.css:98) | **eckig** |
  | `.btn-outline` | document-list.tsx:80 | `var(--radius-md)` (blocks.css:204) | **eckig** |
  | `.download-link` | logo-showcase.tsx:8 | keine | **Text-Link** |

- **Kern:** Drei verschiedene Download-Paradigmen nebeneinander — runder Icon-Button (Block-Cards, Asset-Lib-Karten teils), eckiger Button (Doc-Lib), reiner Text-Link (Logo-Showcase). Nutzer sehen für „dasselbe" (Datei herunterladen) drei Formen.
- **Fix-Vorschlag:** Ein einheitliches Download-Affordance-Muster definieren (z.B. runder Icon-Button `.card-dl` als Standard für Karten, Pille für „Package"-Downloads) und die Doc-Library angleichen. Eigene Komponente `<DownloadButton>` wäre der saubere Weg.
- **Aufwand:** M-L (CSS-Vereinheitlichung + ggf. gemeinsame Komponente)
